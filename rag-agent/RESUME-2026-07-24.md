### A. 행축 재구성 "다른 데이터셋 같은 잣대" 질문 종결 ✅
- **엄밀 잣대(gold tree 대조 exact-match)는 HiTab에만 댈 수 있다.** MultiHiertt/FinQA는
  gold 헤더 트리가 없고(proxy만), AIT-QA는 트리 기제공(재구성 대상 없음), WikiSQL은 flat.
  → HiTab 행 `.56`은 **원리적으로 HiTab 전용 숫자**, 선택이 아니라 계측기(gold tree) 부재.
- **관대 proxy로 맞추면 신호가 포화된다.** HiTab을 MultiHiertt와 동일한 segment-coverage로
  채점하면 **col .943 / row .998** (exact는 .88/.56) — MultiHiertt row `.933`보다도 높음.
  coverage는 *잘못 넣은* segment만 벌하고 *누락된 조상*은 안 벌하는데, HiTab 행축의 지배적
  실패가 바로 누락(얕은 경로)이라 proxy가 못 잡음.
- **결론(인용 규율):** MultiHiertt row `.93`을 "행축 재구성이 잘 됨"의 근거로 **인용 금지.**
  전이 가능한 건 숫자가 아니라 **메커니즘**(행깊이 > stub열수 → 붕괴; expressible=yes .97 vs
  expressible=no .11). 각 데이터셋의 그런 표 비율은 gold 행헤더 트리가 있어야 재는데 없음 =
  **미측정 열린 과제.**
- **재현:** `PYTHONPATH=. python scripts/reconstruct_coverage_vs_exact.py`
  (exact + coverage 병렬 채점, 이번에 추가).

### B. "행 계층을 다룬 논문들" 문헌 지형 (웹 검색)
아무도 bare 격자에서 행을 복원하지 않는다 — **회피하거나 신호를 보존**한다:
- **OHD** (Orthogonal Hierarchical Decomposition, arXiv 2602.01969): 행·열 트리를 독립 유도
  (Orthogonal Tree Induction). 단 **셀 역할(행헤더/열헤더/데이터)이 이미 라벨된 걸 전제** →
  bare 격자 안 다룸. 내 접근과 가장 정면. 축별 정확도 미보고.
- **TUTA** (arXiv 2010.12537): 양방향 좌표 트리, tree-based transformer. 구조 파싱된 걸 소비.
- **HiTab 원 논문**: 병합셀 + **들여쓰기** + 수식에서 규칙 기반 트리 추출 → 들여쓰기 필요.
- **TableSense / Table Header Detection(AAAI)**: 병합·들여쓰기가 불충분할 때 ML로 헤더 계층
  탐지 (진짜 "행 문제를 푸는" 학습 방향).
- 공통: **축별 정확도를 쪼개 보고한 논문이 없다** → 문헌이 행축 약점을 숨김.
- **포지셔닝:** "행을 더 잘 복원한다"로 싸우면 학습기(OHD/TableSense)에 밀림. **"복원된
  계층을 셀:문장 verbalize해 완전성 보장"** 프레이밍 유지가 안전한 해자.

## 지금 돌리고 있던 실험 — 문장화가 정답률로 전환되나 (핵심 기여)

**H6(총합행 주입)은 HiTab 전용 사례연구**였고, 이건 **핵심 기여(verbalization)를 3솔버 ×
공식 EM으로** 검증하는 것. 스크립트: `scripts/verbalize_answer_eval.py` (git-modified).
- arms: `oracle`(천장) / `original_sent`(**우리 문장화 검색**) / `rag_1t1c` / `rag_rowchunk`
- 지표: `hmt_exact_match` (HiTab 공식 채점기, primary) + CI95 + `Δ(original_sent − rag_*)` bootstrap
- 모집단: `hitab_dev_random_sample` — **랜덤샘플이라 H6의 flips-first 편향 없음.**
- 스모크(n=3, 8b) 통과 = 배선·API·공식지표·CI 정상.

### 연구실에서 재개할 명령 (매일, 429로 끊기면 `--resume`이 이어감)
```bash
cd ~/T2/rag-agent   # (경로는 환경에 맞게)
source .venv/bin/activate   # GROQ_API_KEY 환경변수 필요

# 8b (500k TPD — 하루 완주 가능, n=60 × 4 arms ≈ 360k 토큰)
PYTHONPATH=. python scripts/verbalize_answer_eval.py --n 60 \
  --model llama-3.1-8b-instant --resume \
  --out results/verbalize_answer_8b_n60.json

# 70b (100k TPD — 하루 ~20쿼리, 여러 날 resume)
PYTHONPATH=. python scripts/verbalize_answer_eval.py --n 60 \
  --model llama-3.3-70b-versatile --resume \
  --out results/verbalize_answer_70b_n60.json

# gpt-oss-120b (200k TPD)
PYTHONPATH=. python scripts/verbalize_answer_eval.py --n 60 \
  --model openai/gpt-oss-120b --resume \
  --out results/verbalize_answer_gptoss120b_n60.json
```
- **속도 주의:** 스크립트 자체 throttle 기본 `--tpm 6000`이 병목(분당 3–4콜) → 8b 완주 ~50–70분.
  실제 Groq 분당한도가 더 높으면 `--tpm 20000` 정도로 올려 단축 가능(순간 429 위험 감수).
- **읽는 법:** "문장화 → 정답률 전환"은 `Δ original_sent − rag_*`로 본다. 8b는 리더 병목이라
  Δ 작게 나와도 예상된 것 — **70b/gpt-oss에서 벌어지는지가 관전 포인트.**

### 이 세션의 로컬 8b run 상태
- 커밋 시점에 로컬에서 8b n=60이 아직 **진행 중**(결과파일 미생성; 스크립트가 끝날 때 저장).
  연구실에서 위 명령으로 **처음부터 다시 돌리면 됨**(로컬 미완 run에 의존하지 말 것).
