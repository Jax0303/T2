### B. "행 계층을 다룬 논문들" 문헌 지형 (웹 검색)
아무도 bare 격자에서 행을 복원하지 않는다 — **회피하거나 신호를 보존**한다:
- **OHD** (Orthogonal Hierarchical Decomposition, arXiv 2602.01969): 행·열 트리를 독립 유도
  (Orthogonal Tree Induction). 단 **셀 역할(행헤더/열헤더/데이터)이 이미 라벨된 걸 전제** →
  bare 격자 안 다룸. 내 접근과 가장 정면. 축별 정확도 미보고.
- **TUTA** (arXiv 2010.12537): 양방향 좌표 트리, tree-based transformer. 구조 파싱된 걸 소비.
- **HiTab 원 논문**: 병합셀 + **들여쓰기** + 수식에서 규칙 기반 트리 추출 → 들여쓰기 필요.
- **TableSense / Table Header Detection(AAAI)**: 병합·들여쓰기가 불충분할 때 ML로 헤더 계층
  탐지 (진짜 "행 문제를 푸는" 학습 방향).
- 공통: **축별 정확도를 쪼개 보고한 논문이 없다** → 문헌이 행축 약점을 숨김.
- **포지셔닝:** "행을 더 잘 복원한다"로 싸우면 학습기(OHD/TableSense)에 밀림. **"복원된
  계층을 셀:문장 verbalize해 완전성 보장"** 프레이밍 유지가 안전한 해자.

#### B-0. 왜 걔네는 bare 격자에서 행을 복원 안 하나 — 논문별 명시 이유 (웹, 2026-07-24)
넷 다 **"순수 격자 기하학은 논리 계층을 결정하지 못한다"**에 동의. 다른 건 *어떤 추가 신호에
기대느냐*뿐. 내 접근은 그 추가 신호가 **빠진 bare 격자**를 먹으니 행이 무너지는 것 = 문헌이
"여기엔 신호가 필요하다"고 인정한 그 구멍을 신호 없이 **측정**한 것.

| 논문 | 계층이 어렵다는 명시 이유 | 추가로 요구하는 신호 | 회피/해결 방식 |
|---|---|---|---|
| **OHD** (2602.01969) | "병합·오프셋 헤더가 N×M 인덱싱을 깨 **공간 근접이 논리 연관을 보장 못함**, 좌표 스캔 무력" | 셀 역할 라벨(입력) | 이미 라벨됐다고 **가정**(정당화 안 함) |
| **HiTab** (2108.06712) | "병합·들여쓰기·볼드가 계층을 표현" | 병합·들여쓰기·볼드 | 포맷 휴리스틱(94%) |
| **TUTA** (2010.12537) | "표 이해엔 spatial·hierarchical·semantic 정보 필요, flat이 놓침" | 구조 명시 인코딩 | 트리 표현 + 사전학습 |
| **TableSense** (2106.13500, AAAI-19) | "포맷 속성 단순 휴리스틱은 복잡표(금융)에서 **불충분**" | 20개 셀 특징 | CNN 학습 |

- **핵심:** OHD가 "공간 근접 ≠ 논리 연관"이라며 **가정으로 지나간** 구멍을, 나는 실제로 재서
  "얼마나 무너지나(행 exact 0/14,565)"를 보임. → 경쟁 프레임은 "행을 더 잘 복원"이 아니라
  여전히 **"복원 불가 구간까지 포함해 verbalize로 완전성 보장"**.
- **문장 전이 확증(`results/sentence_accuracy_hitab_raw.json`):** 복원 트리로 문장 생성 시
  행경로 오류 ≫ 열경로 오류 (long: 20,886 vs 6,802 = 3.1×; short/med: 10,267 vs 6,791).
  열오류는 스타일 무관 평평(~6,800), 행오류는 leaf→전체경로에서 **2배 폭발** = 조상 붕괴 지문.
  **단 정답률 실험(H6/verbalize_answer_eval)은 아직 GOLD 트리 입력**(`hitab.py:34-35` →
  `top_root`/`left_root`)이라 행 실패가 EM에 미반영 = 복원 트리로 갈아끼워 재측정이 열린 과제.

### B-1. "라벨된 데이터셋 쓰면 내 패러다임과 모순 아니냐" — 종결 ✅
내 패러다임 = **raw 표 → 구조 복원 → 셀 문장화.** 라벨을 써도 모순 아님. **라벨이 어디로
들어가느냐**가 전부다:
- **입력으로 먹으면 = OHD가 됨 (금지).** OHD는 `pre-identified as Row/Column Header/Data`를
  입력으로 받음 → raw 문제를 **안 푼다.** 셀 역할 라벨을 복원기 입력에 넣는 순간 나도 같아짐.
- **채점기(정답지)로만 쓰면 = 정당 (필수).** raw 표를 입력하고, 복원 결과를 gold 헤더 트리와
  **exact-match 대조만.** 지도학습에서 정답 라벨로 정확도 재는 것과 동일. 정확도를 재려면
  라벨된 데이터셋이 **오히려 반드시 필요** (섹션 A "엄밀 잣대는 HiTab에만" = 이 얘기).
- **규칙 한 줄:** gold 트리·역할 라벨은 **평가에만. 절대 복원기 입력으로 넣지 않는다.**
- **진짜 함정 = AIT-QA.** AIT-QA는 트리 기제공(재구성 대상 없음). 그 트리를 **입력으로 쓰면
  그 순간 OHD**가 됨. HiTab처럼 **채점용으로만** 쓰면 안전.
- 검증(웹, 2026-07-24): OHD arXiv `2602.01969` 원문 확인 — OTI 입력이 pre-labeled 역할 전제,
  축별 복원 정확도 미보고. TUTA(2010.12537)=파싱된 좌표 트리 소비. HiTab(2108.06712)=병합·
  들여쓰기 규칙 추출(94%). 셋 다 **bare 격자에서 행 복원 안 함** = 사용자 이해 정확.

### 반드시 알아야 할 교란요인 2가지
1. **oracle(.283) < rag_rowchunk(.317)** — 정답 표를 통째로 줬는데도 행청크 10개보다 낮음.
   즉 8b에겐 **"전체 표를 넣는 컨텍스트 형태" 자체가 병목**(lost-in-the-middle). 검색 문제 아님.
2. **예산이 비대칭 적용됨.** `format_context`는 첫 청크를 예산과 무관하게 통째로 넣음
   (`if used+len > budget and out:` — 첫 청크는 `out`이 비어 통과). 그래서 전체표 arm은
   1200토큰 예산을 **무시하고 최대 4346토큰까지** 들어가고(전체표 35%가 예산 초과), rag_rowchunk만
   1200에서 잘림(70%). **전체표 arm이 오히려 더 많은 컨텍스트를 받고도 짐.**
   → rag_rowchunk와의 비교는 현재 설계상 **동일조건 비교가 아님.**

### 정직한 읽기
- **동일조건 비교는 `original_sent` vs `rag_1t1c`뿐**(둘 다 전체표 1청크, 검색만 다름):
  표 식별 **+18.4pp**(.717 vs .533), EM **+6.7pp**(CI에 0 포함, n=60에선 미달).
  → 문장화 검색이 **표 식별을 개선**한다는 근거는 나옴. 정답률 전환은 n 부족.
- **8b는 천장(oracle .283)이 너무 낮아 방법 판별력이 없음** — 본문 예고대로.
  **판별은 70b/gpt-oss 완주에서** 나옴.
- 다음 설계 결정 필요: rag_rowchunk와 공정 비교하려면 (a) 전체표 arm에도 예산을 강제하거나
  (b) 우리 arm을 셀문장 top-k 컨텍스트로 바꿔 granularity를 맞춰야 함.
