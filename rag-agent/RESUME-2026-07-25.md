**LLM이 개입하지 않는 순수 검색 지표다** (같은 파일 `osc.note`: "OSC over full

→ **양쪽 다 검색 지표라 비교 자체는 정당하다. 라벨만 틀렸다.**

## 1. 오늘 새로 확인한 사실 — .50이 낮은 건 방법 탓이 아니다

### (a) 모집단이 벤치마크에서 제일 어려운 16%였다

`load_queries(split='dev')` 직접 집계 (재현: 아래 스니펫):

```
HiTab dev 전체 1,671 쿼리
  aggregation = none (단순 조회)   1,195   71.5%
  단일 operand (≤1)                1,395   83.5%
  다중 operand 산술                  ~276   16.5%
```

```bash
PYTHONPATH=. python3 -c "
from rag_agent.bench.hitab import load_queries
from collections import Counter
qs,_=load_queries(split='dev')
print(len(qs), Counter(q.aggregation for q in qs).most_common(5))
print(sum(1 for q in qs if len(q.gold_operands or [])<=1))"
```

그런데 `.3953 → .5000`을 잰 모집단은 `hitab_dev_arith_m>=2`
(`results/h6_rerun_20260707/gptoss120b.json` → `population.name`) — **다중 operand 산술만.**
OSC 연구용으로 일부러 고른 하드 슬라이스다.

HiTab 논문들이 표준으로 보고하는 건 dev/test 전체다. **전체로 옮기는 건 체리피킹이 아니라
표준 모집단으로 돌아가는 것**이다. 다만 논문에는 모집단을 명시해야 한다.

### (b) 리더가 병목이다 — 검색이 아니라

`results/verbalize_answer_8b_n60.json` → `summary.oracle`:

```
8b oracle (gold 표를 통째로 줌, 검색 오류 0) = hmt_exact_match .2833
```

**정답 표를 그대로 줬는데 28%.** 검색이 완벽해도 이 숫자다. 같은 조건에서
gpt-oss-120b는 oracle **.60** (`results/verbalize_answer_gptoss120b_n60.json` →
`summary.oracle.hmt_exact_match`, **단 n=5**, CI95 [.2, 1.0] — 방향만 참고).

거기에 `--max-context-tokens 1200` 족쇄가 걸려 있는데, 이건 **순전히 Groq 무료 티어
TPM 6000 때문**이다(`scripts/verbalize_answer_eval.py:74` 주석). 전체 표의 35%가 이
예산을 넘는다(`RESUME-2026-07-24.md` §교란요인 2).

### (c) 아직 한 번도 측정 안 된 조건

| 레버 | 지금까지 | 바꿀 값 |
|---|---|---|
| 모집단 | 산술 다중 operand (하드 16%) | **dev 전체** |
| 리더 | llama-3.1-8b (oracle .2833) | **프론티어 모델** |
| 컨텍스트 예산 | 1200토큰 (35% 잘림) | **해제** |
| 구조 입력 | gold 트리 | 그대로 (교수님 지시와 일치) |

네 개를 동시에 바꾼 조건은 **한 번도 돌린 적이 없다.** 여기서 답변 정확도가 얼마 나오는지가
지금 가장 중요한 미지수다.

### (d) 반드시 같이 붙일 단서 — 그냥 넘어가면 두 번째 사고가 된다

**gold 트리를 입력으로 주는 조건은 오라클 상한선이지, raw 파이프라인 성능이 아니다.**

OHD가 HiTab에서 보고한 값이 **60.07 EM**이다
(`results/h6_rerun_20260707/gptoss120b.json` → `answer_accuracy_official.metric` 주석).
따라서:

- gold 트리 조건에서 .8대가 나오면 → **"gold 구조 제공 시 상한선"으로만 표기.**
  OHD 60.07과 직접 비교하면 안 된다(그쪽은 구조를 안 받음).
- raw 격자 조건 숫자는 별도로, 행축 복원 .5446이 반영된 값으로 따로 보고.

이 구분을 안 하면 0.89 사고가 그대로 반복된다.

## 3. 교수님 관련

### (a) API 예산 요청 — 오늘/내일 중

교수님이 직접 **"LLM은 좋은 거 쓰라"**고 하셨으므로 명분은 확보돼 있다. 위 §1(b)가
근거다: 8b는 gold 표를 줘도 oracle .2833이라 **방법 판별력이 없다.** 무료 티어로는
70b/gpt-oss가 n=4~5에서 끊긴다.

### (b) 행축 질문 답변 카드 — 이미 측정 끝남

교수님 질문: *"열 병합 데이터가 표현돼 있다면 행 구조도 비슷한 방식으로 복원 가능하지 않나?"*

이미 병합셀 정보(`merged_regions`)를 실제로 먹여봤다.
`results/tree_reconstruct_hitab_raw_merged.json`:

```
col_path_exact_match = .9428
row_path_exact_match = .5446     ← 병합 정보를 줘도 행은 안 올라감
```

그리고 **왜 안 되는지가 같은 파일에 정량화돼 있다** (`row_axis_by_depth_expressible`):

```
expressible = yes (행깊이 ≤ stub 열 수)  → row exact .9454  (n_paths 3,774)
expressible = no  (행깊이 >  stub 열 수)  → row exact .1026  (n_paths 3,422)
```

**즉 알고리즘이 약한 게 아니라, 격자에 정보가 물리적으로 없는 구간이다.**
행 계층은 병합이 아니라 **들여쓰기**로 표현되는데(HiTab 원논문 2108.06712도 들여쓰기를
씀), HiTab export에는 들여쓰기가 빠져 있다.

→ **교수님 직관은 절반 맞다.** 병합으로 표현 가능한 표에선 실제로 .9454가 나온다 —
열축(.9428)과 동급이다. 안 되는 건 나머지 절반이고, 그게 말씀하신 "인텐트 많은 표"와
같은 집단이다. `by_max_depth`도 같은 얘기를 한다: row_depth1 .8858 → depth2 .6024 →
depth3 .3275.

이건 "해보겠습니다"가 아니라 **"해봤고 이 조건에서 .945, 저 조건에서 .103, 원인은
신호 부재"**로 답할 수 있는 카드다.

### (c) 방어되는 것 — "GOLD 문장 생성 100%"

`results/sentence_accuracy_hitab.json` → `by_style.*`:
f1 = **.9981** (67,315 gold 셀 중 오류 128개, 전부 `err_wrong_col_path`;
`err_wrong_value` 0, `err_wrong_row_path` 0, `dropped_gold_cells` 0).
크기 10–20 버킷은 정확히 **1.0**. 반올림해서 100%라 한 건 문제없다.
