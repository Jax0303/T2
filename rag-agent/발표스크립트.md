# 랩미팅 발표 스크립트 (말로 풀어서)

영어 용어 빼고, 평어체로 그대로 읽으면 발표가 되는 형태입니다.

---

## 교수님 지시

> **"원본이랑 벡터DB 따로 두고, 어떤 정보가 필요할 때 어떻게 검색해야 답이 잘 나오는지, 수치로 확인해와라"**

이 지시에 대해 제가 한 일과 결과를 순서대로 말씀드리겠습니다.

---

## 1. 우선 두 저장소의 **역할**부터 분리했습니다

같은 표를 두 곳에 다르게 넣어뒀어요. 한쪽은 **벡터 검색 인덱스**예요.
표를 글자로 풀어서 임베딩 벡터로 바꿔놓은 거고, "이 질문이랑 비슷한
표가 뭐냐"를 빠르게 추리는 용도입니다. 다른 한쪽은 **원본 2차원 구조
그대로**예요. 행·열·헤더 트리·셀 값이 다 살아있는 형태로, "이 표의
'총계' 행에 있는 '2017년' 컬럼 값이 정확히 얼마냐"를 묻는 용도입니다.

**제가 만든 원칙은 둘을 섞지 않는다**입니다. 벡터DB는 후보를 발굴할
때만 쓰고, 원본 구조는 그 후보가 정말 맞는지 검증하고 계산이 필요할
때 값을 정확히 뽑을 때만 씁니다. 이전 연구(HART)는 두 신호를 가중평균
으로 섞었는데 효과가 없었어요. 그래서 이번엔 단계를 분리했습니다.

---

## 2. 질문 종류에 따라 **거치는 단계가 달라지게** 만들었습니다

질문이 들어오면 먼저 어떤 유형인지 분류해요. HiTab 논문 부록에 있는
6가지 분류를 그대로 따랐습니다:

- 단순 조회 (특정 셀 값 묻기)
- argmax류 ("어느 지역이 가장 높냐")
- 비교/개수 ("A가 B보다 큰가", "몇 개냐")
- 산술 (합·차·평균·비율)
- 다단계 수식 (`=B20+B21+B22` 같이 셀 여러 개 조합)
- 추론만 (표가 필요 없는 일반 상식 질문)

분류 결과에 따라 단계를 갈아끼웁니다. 추론만 필요한 질문이면 검색
단계 자체를 건너뜁니다. 산술 질문이면 LLM한테 직접 계산시키지 않고
**원본 구조에서 셀을 뽑아서 코드가 직접 계산**합니다.

---

## 3. **수치로 확인한** 결과를 말씀드리면

같은 HiTab 데이터셋의 어려운 쿼리 40개로 측정했습니다. 비교 기준은
우리 랩의 이전 시스템(Sidecar)이 같은 데이터에서 받았던 점수예요.

**검색 단계 결과**: 원본 구조로 검증을 한 번 더 거치니까 1순위 정확도가
**57.5% → 67.5%**로 10%p 올랐어요. 5순위까지 보면 **87.5%**로 정답
표를 거의 다 찾아냅니다.

**답변 단계 결과**: 이전 베이스라인이 25%였는데, 이번 시스템이
**47.5%**까지 나왔습니다. **약 22%p 상승**입니다. 신뢰구간으로 보면
32.5%~62.5% 사이로, 베이스라인 25%가 신뢰구간 밖에 있어서 통계적으로도
의미 있는 차이입니다.

**산술 질문 처리 결과**: 비교·개수 질문에서는 8개 중 3개를 **LLM의
계산 능력에 의존하지 않고 코드로만 풀어냈습니다**. 코드 계산이 작동하는
게 수치로 확인된 거죠.

---

## 4. **검증의 효과를 격리**해서 보여드릴 수 있게 ablation도 돌렸습니다

검증 단계를 빼고 똑같이 돌렸더니 답변 정확도가 47.5% → 35%로
떨어졌습니다. **검증 단계가 12.5%p 정도 기여하고 있다**는 게 직접적인
증거예요. 같은 데이터, 같은 모델, 검증만 켜고/껐을 때 차이입니다.

다만 **모든 클래스에서 좋아지지는 않았어요**. 다단계 수식 질문에서는
검증이 오히려 1순위를 12.5%p 떨어뜨렸습니다. 이유는 수식 질문은 "총합"
"비율" 같은 일반 단어를 많이 써서, 검증의 키워드 매칭이 엉뚱한 표를
위로 올리거든요. 이건 **솔직히 단점으로 보고드릴 부분**입니다. 질문
종류마다 검증 가중치를 다르게 주면 고칠 수 있을 거 같다고 말씀드리려고요.

---

## 5. 결과가 **운이 아닌지**도 확인했습니다

샘플링 시드를 0, 1, 2로 세 번 다르게 돌렸어요. 답변 정확도가 각각
47.5%, 40%, 37.5%로 나와서 평균 41.7%입니다. 시드 하나만 골라서 좋은
값을 가져온 게 아니라, **어떤 시드를 써도 이전 베이스라인 25%보다는
확실히 위**라는 게 확인됐습니다.

---

## 6. 무료 LLM 제약은 지켰습니다

선생님이 무료 모델만 쓰라고 하셔서, 로컬에서 돌리는 Qwen 2.5의 7B
모델과 Groq라는 곳의 무료 API 두 가지로만 실험했습니다. OpenAI 같은
유료 API는 한 번도 안 썼어요. **3060Ti 한 장에서 다 돌아가는** 구성입니다.

---

## 7. 한계도 정직하게 말씀드릴 부분

- 다단계 수식(`=(B+C+D)/E` 같은 거) 정확도가 0~12.5%입니다. 검증·검색은
  잘 되는데, **LLM이 어느 셀을 뽑아야 하는지 못 골라요**. 더 큰 70B
  모델을 추출용으로만 써봤더니 비교·산술 클래스는 회복됐지만 다단계
  수식은 그대로 0%였습니다. **이건 모델 자체의 한계**라는 게 우리
  데이터로 확인됐습니다.
- 샘플 크기가 40이라 각 클래스가 8개씩이에요. 단일 클래스의 변동성은
  커요. 다음 실험에서 클래스당 25개씩으로 확장하면 더 단단해질 것
  같습니다.

---

## 한 줄 요약

> **"벡터 인덱스는 후보 찾기에만 쓰고, 원본 2차원 표는 검증과 계산에만
> 쓰는 식으로 역할을 분리했습니다. 그랬더니 같은 무료 모델로도 답변
> 정확도가 25%에서 47.5%로 올랐고, 검증 단계가 12.5%p 기여한다는 것도
> ablation으로 확인했습니다. 다단계 수식 문제만 모델 한계로 남아있고,
> 그 외 클래스는 다 잘 됩니다."**

## 예상 질문 대응

**Q. 그 +10%p가 진짜 검증 덕분이라고 어떻게 확신해?**
- ablation 돌려봤습니다. 검증만 빼면 답변 정확도가 47.5% → 35%로 떨어집니다.
  같은 모델, 같은 데이터, 같은 시드. 검증 켜고/끄는 것만 바꿨어요.

**Q. 40개로는 부족하지 않냐?**
- 부족하다고 봅니다. 그래서 신뢰구간을 같이 계산했어요. 답변 정확도
  47.5%의 95% 신뢰구간이 32.5%~62.5%인데, 베이스라인 25%가 신뢰구간
  밖에 있어서 통계적으로는 유의합니다. 다음에 클래스당 25개로 늘리려고
  합니다.

**Q. 시드 운 아니냐?**
- 시드 3개 돌렸습니다. 47.5%, 40%, 37.5%. 가장 낮은 값도 베이스라인보다는
  높아요.

**Q. 다단계 수식 0%는 너희 코드 문제 아니냐?**
- 처음엔 그렇게 의심해서 trace를 다 까봤어요. 버그 4개 찾아서 다 고쳤습니다.
  고친 후에도 0~12.5%인데, 이건 LLM이 어느 셀을 뽑아야 할지 모르는 거예요.
  코드 계산은 잘 돌아갑니다. 70B 모델로 추출만 돌려봐도 이 클래스는
  회복이 안 됐어요. 모델 자체의 한계입니다.

**Q. 다른 벤치마크(TabFact, FeTaQA)는 안 해봤냐?**
- HiTab에 집중했습니다. 이전 연구(Sidecar)에서 TabFact는 검증이 오히려
  해롭다고 나왔거든요. 위키피디아 표는 구조가 균일해서 키워드로 구분이
  안 됩니다. HiTab 같은 계층형 표에 특화된 결과라고 보고드릴 수 있습니다.

**Q. 코드 어떻게 짰는지 자세히 설명해봐**
- 코드 자체보다는 결정한 정책을 말씀드릴게요. 두 저장소를 분리한 것,
  질문 종류로 단계를 갈아끼우는 것, 산술은 LLM이 아니라 코드가 푸는 것
  세 가지가 핵심입니다. 자세한 구현은 깃 리포지토리 EXPERIMENTS.md에
  정리해뒀습니다.
