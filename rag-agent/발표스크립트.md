# 랩미팅 발표 스크립트 (말로 풀어서)

영어 용어 빼고, 평어체로 그대로 읽으면 발표가 되는 형태입니다.

---

## 교수님 지시

> **"원본이랑 벡터DB 따로 두고, 어떤 정보가 필요할 때 어떻게 검색해야 답이 잘 나오는지, 수치로 확인해와라"**

이 지시에 대해 제가 한 일과 결과를 순서대로 말씀드리겠습니다.

---

## 1. 우선 두 저장소의 **역할**부터 분리했습니다

같은 표를 두 곳에 다르게 넣어뒀어요. 한쪽은 **벡터 검색 인덱스**예요.
표를 글자로 풀어서 임베딩 벡터로 바꿔놓은 거고, "이 질문이랑 비슷한
표가 뭐냐"를 빠르게 추리는 용도입니다. 다른 한쪽은 **원본 2차원 구조
그대로**예요. 행·열·헤더 트리·셀 값이 다 살아있는 형태로, "이 표의
'총계' 행에 있는 '2017년' 컬럼 값이 정확히 얼마냐"를 묻는 용도입니다.

**제가 만든 원칙은 둘을 섞지 않는다**입니다. 벡터DB는 후보를 발굴할
때만 쓰고, 원본 구조는 그 후보가 정말 맞는지 검증하고 계산이 필요할
때 값을 정확히 뽑을 때만 씁니다. 이전 연구(HART)는 두 신호를 가중평균
으로 섞었는데 효과가 없었어요. 그래서 이번엔 단계를 분리했습니다.

---

## 2. 질문 종류에 따라 **거치는 단계가 달라지게** 만들었습니다

질문이 들어오면 먼저 어떤 유형인지 분류해요. HiTab 논문 부록에 있는
6가지 분류를 그대로 따랐습니다:

- 단순 조회 (특정 셀 값 묻기)
- argmax류 ("어느 지역이 가장 높냐")
- 비교/개수 ("A가 B보다 큰가", "몇 개냐")
- 산술 (합·차·평균·비율)
- 다단계 수식 (`=B20+B21+B22` 같이 셀 여러 개 조합)
- 추론만 (표가 필요 없는 일반 상식 질문)

분류 결과에 따라 단계를 갈아끼웁니다. 추론만 필요한 질문이면 검색
단계 자체를 건너뜁니다. 산술 질문이면 LLM한테 직접 계산시키지 않고
**원본 구조에서 셀을 뽑아서 코드가 직접 계산**합니다.

---

## 3. **수치로 확인한** 결과를 말씀드리면

같은 HiTab 데이터셋의 어려운 쿼리 40개로 측정했습니다. 비교 기준은
우리 랩의 이전 시스템(Sidecar)이 같은 데이터에서 받았던 점수예요. 이전
시스템 코드 자체는 이번에 레포를 정리하면서 빼고, **비교에 쓴 베이스라인
점수 파일만 `rag-agent/results/baselines/sidecar_cot_baseline.json`에
남겨뒀습니다**. 그래서 NM 25% 출처가 깃에 그대로 박혀있어요.

**검색 단계 결과**: 원본 구조로 검증을 한 번 더 거치니까 1순위 정확도가
**57.5% → 67.5%**로 10%p 올랐어요. 5순위까지 보면 **87.5%**로 정답
표를 거의 다 찾아냅니다.

**답변 단계 결과**: 이전 베이스라인이 25%였는데, 이번 시스템이
**47.5%**까지 나왔습니다. **약 22%p 상승**입니다. 신뢰구간으로 보면
32.5%~62.5% 사이로, 베이스라인 25%가 신뢰구간 밖에 있어서 통계적으로도
의미 있는 차이입니다.

**산술 질문 처리 결과**: 비교·개수 질문에서는 8개 중 3개를 **LLM의
계산 능력에 의존하지 않고 코드로만 풀어냈습니다**. 코드 계산이 작동하는
게 수치로 확인된 거죠.

---

## 4. **검증의 효과를 격리**해서 보여드릴 수 있게 ablation도 돌렸습니다

검증 단계를 빼고 똑같이 돌렸더니 답변 정확도가 47.5% → 35%로
떨어졌습니다. **검증 단계가 12.5%p 정도 기여하고 있다**는 게 직접적인
증거예요. 같은 데이터, 같은 모델, 검증만 켜고/껐을 때 차이입니다.

다만 **모든 클래스에서 좋아지지는 않았어요**. 다단계 수식 질문에서는
검증이 오히려 1순위를 12.5%p 떨어뜨렸습니다. 이유는 수식 질문은 "총합"
"비율" 같은 일반 단어를 많이 써서, 검증의 키워드 매칭이 엉뚱한 표를
위로 올리거든요. 이건 **솔직히 단점으로 보고드릴 부분**입니다. 질문
종류마다 검증 가중치를 다르게 주면 고칠 수 있을 거 같다고 말씀드리려고요.

---

## 5. 결과가 **운이 아닌지**도 확인했습니다

샘플링 시드를 0, 1, 2로 세 번 다르게 돌렸어요. 답변 정확도가 각각
47.5%, 40%, 37.5%로 나와서 평균 41.7%입니다. 시드 하나만 골라서 좋은
값을 가져온 게 아니라, **어떤 시드를 써도 이전 베이스라인 25%보다는
확실히 위**라는 게 확인됐습니다.

---

## 6. 무료 LLM 제약은 지켰습니다

선생님이 무료 모델만 쓰라고 하셔서, 로컬에서 돌리는 Qwen 2.5의 7B
모델과 Groq라는 곳의 무료 API 두 가지로만 실험했습니다. OpenAI 같은
유료 API는 한 번도 안 썼어요. **3060Ti 한 장에서 다 돌아가는** 구성입니다.

레포 정리도 같이 했습니다. 예전 thesis 단계 코드(§3 직렬화 진단,
§4 임베더 프로빙, §5 HART)는 다 빼고 **이번 작업(§6) 단독 패키지**로
만들었어요. 베이스라인 점수 파일(NM 25% 출처)만 따로 보존했고, 누가
처음 클론해도 `pip install -e rag-agent/` 한 줄로 돌아갑니다.

## 7. 한계도 정직하게 말씀드릴 부분

- 다단계 수식(`=(B+C+D)/E` 같은 거) 정확도가 0~12.5%입니다. 검증·검색은
  잘 되는데, **LLM이 어느 셀을 뽑아야 하는지 못 골라요**. 더 큰 70B
  모델을 추출용으로만 써봤더니 비교·산술 클래스는 회복됐지만 다단계
  수식은 그대로 0%였습니다. **이건 모델 자체의 한계**라는 게 우리
  데이터로 확인됐습니다.
- 샘플 크기가 40이라 각 클래스가 8개씩이에요. 단일 클래스의 변동성은
  커요. 다음 실험에서 클래스당 25개씩으로 확장하면 더 단단해질 것
  같습니다.

## 예상 질문 대응

**Q. 그 +10%p가 진짜 검증 덕분이라고 어떻게 확신해?**
- ablation 돌려봤습니다. 검증만 빼면 답변 정확도가 47.5% → 35%로 떨어집니다.
  같은 모델, 같은 데이터, 같은 시드. 검증 켜고/끄는 것만 바꿨어요.

**Q. 40개로는 부족하지 않냐?**
- 부족하다고 봅니다. 그래서 신뢰구간을 같이 계산했어요. 답변 정확도
  47.5%의 95% 신뢰구간이 32.5%~62.5%인데, 베이스라인 25%가 신뢰구간
  밖에 있어서 통계적으로는 유의합니다. 다음에 클래스당 25개로 늘리려고
  합니다.

**Q. 시드 운 아니냐?**
- 시드 3개 돌렸습니다. 47.5%, 40%, 37.5%. 가장 낮은 값도 베이스라인보다는
  높아요.

**Q. 다단계 수식 0%는 너희 코드 문제 아니냐?**
- 처음엔 그렇게 의심해서 trace를 다 까봤어요. 버그 4개 찾아서 다 고쳤습니다.
  고친 후에도 0~12.5%인데, 이건 LLM이 어느 셀을 뽑아야 할지 모르는 거예요.
  코드 계산은 잘 돌아갑니다. 70B 모델로 추출만 돌려봐도 이 클래스는
  회복이 안 됐어요. 모델 자체의 한계입니다.

**Q. 다른 벤치마크(TabFact, FeTaQA)는 안 해봤냐?**
- HiTab에 집중했습니다. 이전 연구(Sidecar)에서 TabFact는 검증이 오히려
  해롭다고 나왔거든요. 위키피디아 표는 구조가 균일해서 키워드로 구분이
  안 됩니다. HiTab 같은 계층형 표에 특화된 결과라고 보고드릴 수 있습니다.

**Q. 레포 구조가 예전이랑 다른데?**
- 예전엔 §3 직렬화 진단, §4 임베더 프로빙, §5 HART, §6 이번 작업이
  같이 있었는데, 이번에 최종 단계로 정리하면서 **§6만 남기고 다 뺐습니다**.
  HART 코드는 의존성이 있어서 HiTab 로더만 패키지 안으로 옮기고 나머지는
  전부 삭제했어요. 비교에 쓴 베이스라인 JSON 한 개만 보존했습니다.
  지금은 `rag-agent/` 하나만 있고, `pip install -e rag-agent/`로 바로
  설치돼요. 39만 줄 정도 삭제했습니다.

**Q. 코드 어떻게 짰는지 자세히 설명해봐**
- 코드 자체보다는 결정한 정책을 말씀드릴게요. 두 저장소를 분리한 것,
  질문 종류로 단계를 갈아끼우는 것, 산술은 LLM이 아니라 코드가 푸는 것
  세 가지가 핵심입니다. 자세한 구현은 깃 리포지토리 EXPERIMENTS.md에
  정리해뒀습니다. (필요하시면 아래 "코드 흐름 상세" 섹션으로 답변
  가능합니다)

## 부록 — 코드 흐름 상세 (필요할 때만 꺼내쓰기)

교수님이 "코드 어떻게 짰는지 자세히" 물으시면 아래 7단계를 차례로
설명드리면 됩니다. 함수 이름까지 다 외울 필요는 없고, **순서와 각
단계에서 일어나는 일**만 기억하시면 됩니다.

### 진입점

`RAGAgent.run(query)` 한 함수가 전체 오케스트레이션. 들어온 쿼리를
1번부터 7번까지 단계별로 통과시키고, 각 단계의 입출력을 trace 객체에
다 기록합니다. 단계를 건너뛰면 그 사실도 trace에 남깁니다.

### 1단계 — 질문 의도 분류

`classify_query(query)` 함수가 규칙 기반으로 분류해요. 미리 정의해둔
정규식 패턴 6개를 순서대로 검사:

1. 수학 기호 ≥2개 (`+`, `-`, `*`, `/`가 두 번 이상) → **다단계 수식**
2. 산술 키워드 ≥2개 (예: "average"와 "difference" 동시 등장) → **다단계 수식**
3. entity-cue 패턴 ("who/which … higher/lower …") → **argmax류** (이름 답)
4. 단일 산술 키워드 ("sum of", "ratio", "percentage" 등) → **산술**
5. argmax 키워드 ("highest", "most" 등) → **argmax류**
6. 비교 키워드 ("greater than", "more than" 등) → **비교**
7. 어디에도 안 걸리면 → **단순 조회**

여기서 `QueryIntent`라는 작은 자료구조를 만들어서 다음 단계로 넘깁니다.

### 2단계 — 단계 계획

`plan_stages(intent)` 함수가 의도에 따라 **어떤 단계를 실제로 돌릴지**
미리 결정:

- 추론만 필요 → 빈 리스트 (검색·검증 다 스킵)
- 단순 조회 / argmax / 비교 → 검색 + 검증 + LLM 리더
- 산술 / 다단계 수식 → 검색 + 검증 + **코드 계산** + LLM 리더 (백업)

이 계획표를 들고 3단계부터 단계별 실행 여부를 if문으로 체크합니다.

### 3단계 — 벡터 검색

`VectorStore.search()`에서 세 일이 일어남:

1. **임베딩**: bge-large 모델에 쿼리 문자열을 넣어 1024차원 벡터 1개 생성
2. **Chroma 쿼리**: 코사인 유사도 top-20 청크를 가져옴 (거리 점수 포함)
3. **테이블 단위 집계**: 한 테이블에 여러 청크가 걸릴 수 있으니
   `table_id`별로 묶고, 가장 가까운 점수만 남김. 상위 5개 테이블 반환

여기까지는 원본 2D 구조를 **한 번도 안 봤습니다**. 벡터 인덱스만으로
후보 5개를 추렸어요.

### 4단계 — 원본과 교차 검증

`rerank(query, hits, original_store, w_vector=0.7, w_verify=0.3)` 함수가
벡터로 받은 후보 5개를 들고 **각각 원본 저장소에서 따로 꺼내** 점수를
다시 매깁니다. 후보 1개당:

1. `original_store.get(table_id)` → 파싱된 2D 표 객체
2. 쿼리에서 키워드와 숫자를 정규식으로 뽑음
3. 그 표의 모든 헤더 경로에서 단어 집합 만들기
4. 그 표의 모든 셀에서 숫자 집합 만들기
5. **키워드 겹침** + **숫자 겹침** 계산
6. `confidence = 0.6 × keyword + 0.4 × number` (쿼리에 숫자 없으면 keyword만)
7. **최종 점수**: `0.7 × vector_score + 0.3 × confidence`로 재정렬

여기가 **원본 저장소가 처음 쓰이는 지점**. 벡터DB가 "비슷한 표 5개"
했고, 원본은 "이 5개 중 진짜 질문의 단어·숫자를 가진 표는 뭐냐"로 검증.

### 5a단계 — 코드 계산 (산술 쿼리만)

검증 끝나서 top-1 테이블이 정해졌어요. 산술 쿼리면 두 단계:

#### (a) LLM에게 JSON 추출 시키기

`extract_plan(llm, query, top_table)` 함수가 LLM에 다음을 보냄:

- 시스템: "JSON으로만 답해라. 스키마는 `{cells:[{var, row_header, col_header}], expression}` 이다. 사칙연산만 써라. 셀 식별은 행/열 헤더 경로로 해라"
- 유저: 표 렌더링 + 질문 텍스트

LLM 출력에서 JSON 블록을 정규식으로 찾아 파싱. 실패하면 빈 plan 반환.

#### (b) 셀 해석 + 안전 평가

`evaluate_plan(plan, table)` 함수가 각 셀에 대해:

1. `OriginalTable.resolve(row_header, col_header)` 호출
   - 모든 행/열의 헤더 경로를 `" :: "`로 조인
   - 토큰을 split해서 각 토큰이 **word boundary 기준 substring**으로 존재하는지 검사
   - 후보가 여러 개면 헤더 경로 글자 수가 가장 긴 것 = 가장 구체적인 매치 선택
2. (row, col, value) 얻고 value를 float으로 변환
3. `env = {"x1": 18.7, "x2": 15.4, "x3": 21.7}` 같은 딕셔너리 완성

그 다음 expression(`"(x1 + x2 + x3)"`)을:

4. `ast.parse(expr, mode="eval")`로 문법 트리 만듦
5. 직접 만든 walker로 트리를 순회. **허용된 노드만 처리**:
   - `Constant` (숫자), `Name` (env 찾기), `BinOp` (+−×÷), `UnaryOp` (+, −)
   - 그 외 (`Call`, `Attribute`, `Import` 등) 만나면 ValueError
6. 결과 float 반환

이 과정에서 Python의 `eval()` 함수는 **단 한 번도 호출되지 않습니다**.
`__import__("os").system(...)` 같은 공격 시도가 들어와도 안 돌아가요.

#### (c) 채택 게이트

```python
op_count = sum(ch in "+-*/" for ch in expression)
adopt = symbolic_ok and (op_count >= 2
        or (intent=="arithmetic_agg" and ops>=1 and cells>=2))
```

연산자 2개 이상이거나, 산술 의도 + 1개 연산자 + 2개 이상 셀이면 채택.
안 그러면 trace에만 기록하고 LLM 리더로 넘김.

이 게이트가 없었을 때 LLM이 `x1` 같은 1셀 plan을 뱉으면 reader 답을
덮어쓰는 부작용이 있었어요.

### 5b단계 — LLM 리더

코드 계산이 채택 안 됐거나 산술 쿼리가 아니면 LLM이 표 보고 직접 답:

1. 검증된 top-1 테이블을 텍스트로 렌더링 (제목 + 컬럼 헤더 경로 + 행 데이터)
2. 시스템 프롬프트: "표만 보고 답해라. `Reasoning:` → `Final answer:`
   형식 써라. 숫자면 단위 빼고 숫자만"
3. LLM 호출 (Qwen 7B 4bit 또는 Groq API)
4. 출력 텍스트에서 정규식 `final\s*answer\s*[:\-]\s*(.+)`로 마지막 답 추출

### 6단계 — 결과 + trace 반환

`AgentResult` 객체에:

- 쿼리, 의도, 계획
- 벡터 top-5, 검증 후 top-5
- 코드 계산 plan + 해석된 셀 + AST 값 + 채택 여부
- 리더 raw output + 추출된 답
- 최종 답변, 출처(`"symbolic"` / `"reader"`)
- 경과 시간

이 trace가 그대로 JSON으로 저장돼서 나중에 지표 다시 계산하거나 실패
케이스 분석할 수 있어요.

### 한 쿼리 끝까지 따라가기 (실제 예시)

쿼리: *"what is the percentage of southern asia, southeast asia and east
asia consisting of economic immigrants?"*

1. 분류 → 키워드 "percentage" 매치 → **arithmetic_agg**
2. 계획 → `[retrieve, verify, symbolic, llm_answer]`
3. 검색 → 벡터 top-5 = `[2793, 2581, 208, 2658, 755]`. 정답 2793이 1위
4. 검증 → 키워드/숫자 overlap 재정렬 → `[2793, 208, 2581, 755, 2658]`
5a. 코드 계산:
    - LLM이 JSON으로 cells 3개 + expression `(x1+x2+x3)` 반환
    - resolve 3번 호출 → 행 15, 16, 17 / 열 0 / 값 18.7, 15.4, 21.7
    - AST eval → 55.8
    - 게이트: 연산자 2개 ≥ 2 → **채택**
6. 결과: 답 "55.8", 출처 "symbolic", 전 단계 trace 다 저장

### 핵심 디자인 결정 3가지 (가장 받기 쉬운 질문)

1. **두 저장소를 분리한 이유**: 벡터는 의미 검색에 강하지만 셀 값
   정확도엔 약함. 원본은 셀 값엔 정확하지만 의미 검색엔 못 씀. 같은
   score로 섞으면 둘 다 약해지는 방향이라 단계로 분리.
2. **LLM에 산술을 안 시키는 이유**: 기존 벤치에서 multi_op 0%였음.
   LLM이 셀은 잘 골라도 곱셈/나눗셈을 자주 틀림. 그래서 LLM은 "어떤
   셀을 어떻게 묶을지"만 결정하고 계산은 코드가.
3. **`eval()` 안 쓰는 이유**: LLM이 만든 문자열을 그대로 실행하면 위험.
   AST 화이트리스트로 사칙연산만 허용. `__import__("os").system(...)`
   들어와도 안 돌아감.
