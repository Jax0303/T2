문장 생성 정확도 측정 결과

셀 하나를 “캡션 + 열 경로 + 행 경로 + 값” 문장으로 굽는 단계의 정확도. 채점은 생성 문장과 정답 문장의 exact match이며, 표 하나가 데이터 셀 수만큼 문장을 내므로 P와 R이 각각 없는 문장을 지어냈는지 / 있어야 할 문장을 빠뜨렸는지를 따로 잡는다.

표 크기별 결과

n = 540개 표, 67,315개 데이터 셀. 오류는 전량 가장 큰 버킷에 몰려 있고, 나머지 두 버킷은 완전무결하다.

변별력이 없었던 두 축

실험은 세 축으로 측정했지만, 그중 둘은 결과가 완전히 동일했다. 보고할 때 축으로 내세우면 오히려 오해를 부른다.

문장 스타일

short = medium = long

세 스타일 모두 .9981 / 오류 128건으로 완전 동일. 채점이 (행 경로, 열 경로, 값) 삼중항 기준이라 표층 문구 길이는 exact match에 영향을 주지 않는다. 스타일은 정확도가 아니라 검색 성능에서 갈릴 변수다.

헤더 경계 입력

추정 = 정답 경계

경계를 guess_n_header_rows로 추정한 파이프라인과 정답 경계를 넣은 대조군이 수치까지 동일. 경계 추정 정확도가 1.000이라 오차를 하나도 더하지 않는다 — 추정 파이프라인이 이미 천장에 붙어 있다.

후자가 실험의 핵심 결과다. 보통은 “정답 경계를 주면 더 잘 나온다”는 격차가 남는데, 여기서는 그 격차가 0이므로 남은 오류를 경계 추정 탓으로 돌릴 수 없다.

잔여 오류의 정체

128건은 모두 같은 실패 유형이다. 실제 출력 예시:

생성 In results by county, lyndon baines johnson democratic, % for sandoval is 3332.
정답 In results by county, lyndon baines johnson democratic for sandoval is 3332.

열 경로에 % 한 마디가 더 붙었다. 병합된 부모 밑의 빈칸이 “자식이 계속된다”인지 “자식이 없다”인지 평면 격자에는 정보가 없어서 생기는 정보이론적 한계이며, 휴리스틱으로 줄일 수는 있어도 원리적으로 0이 되지 않는다. colspan/rowspan 마크업을 보존하는 입력에서만 제거 가능 — 7월 데드라인 기준 향후 과제로만 남긴다.

주의 — 문장 점수로 트리 정확도를 말할 수 없다

Table 1의 F1은 “만들어진 문장이 정확한가”이지 “트리가 정확한가”가 아니다. 두 실험은 채점하는 격자가 다르다.

트리 실험

모든 데이터 칸을 placeholder "1"로 채운 격자를 쓴다 (tree_reconstruct_hitab.py). 칸이 전부 차 있으므로 모든 행의 경로가 채점된다.

문장 실험

진짜 값을 넣고, 빈 칸에는 문장을 만들지 않는다 (if not _nonempty(...): continue). 값이 하나도 없는 행의 경로가 틀려도 문장이 생성되지 않아 채점되지 않는다.

그 결과가 10-20 버킷이다: 트리 행 경로는 .9997로 오류가 있는데 문장 F1은 1.000이다. 그래서 Table 2의 “행 경로 오류 0건”도 행 축이 완벽하다는 뜻이 아니라, 값이 있는 행에서는 틀린 적이 없다는 뜻이다.

따라서 “열 축이 약하다”고 말할 수 없다

문장 오류가 128 : 0이라 열 축 문제로 보이지만, 경로 단위로 보면 두 축은 거의 같은 횟수로 실패한다. 재구성은 단일 carry 루틴 (_hierarchical_carry)을 입력만 전치해 두 축에 똑같이 적용하므로, 축별 로직 자체가 없다.

Axis Paths scored Wrong paths Path exact Cells per fault Sentence errors
Column 4,413~4.9991 16.6128

두 가지가 겹쳐 128 : 0이 된다. ① 이 표들은 평균 세로 16.6 × 가로 8.2로 세로가 길어, 열 경로 하나가 망가지면 행 경로 하나보다 약 2배 많은 셀을 오염시킨다. ② 행 쪽 오류 ~4개는 하필 값이 없는 행에 떨어져 문장이 생성되지 않았고, 그래서 문장 지표에 잡히지 않았다 (트리 지표는 placeholder로 채운 격자를 보므로 잡아낸다).

다만 이것이 부풀린 수치라는 뜻은 아니다. 색인에 실제로 들어가는 것은 문장뿐이므로, “색인되는 문장의 정확도”라는 주장에는 이 지표가 옳다. 틀리는 것은 측정이 아니라 해석이다. 발표·논문에서 .9981은 문장 정확도로만 쓰고, 트리 정확도는 .9991/.9996을 따로 인용할 것. 그리고 잔여 오류를 “열 축 문제”로 귀속하지 말 것.
