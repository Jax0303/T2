## 7. 아키텍처 다이어그램 (`c:/Users/user/Desktop/아키텍처.drawio`, 4페이지)

- **페이지-1** (기존): 구조는 코드와 일치 → 형식만 수정(화살촉 통일, Table→Tree 연결,
  "Tree Reconstruction" 라벨).
- **트리 재구성** (신규): 원시 격자 → 경계 추정(괄호 주석 가드 포함) → carry-fill →
  경로 → 트리. 루트 하나(Table) 아래 columns/rows 두 가지 = **표 1개당 트리 1그루**.
- **문장 생성** (신규): 셀 분해(캡션/열 경로/행 경로/값) → 템플릿 → 문장 → Vector DB.
- **복잡한 표** (신규): 열 3층 × 행 2층 예시, carry 리셋 지점, 반복 라벨(Amount/%)의
  유일 주소.

### 확인한 개념 (질문 답변)

- **문장 생성 방식**: "캡션+행 헤더+열 헤더를 셀에 결합" 이해가 맞음. 정확한 템플릿:
  `In {title}, {열 경로} for {행 경로} is {값}.` — 헤더는 리프가 아닌 **부모 포함 전체
  경로**, 셀 **값** 결합, (table_id, row, col) 메타데이터 저장.
- **검색 시 트리 사용 여부**: 검색 때 트리를 타지 않음. 트리는 색인 시 주소 텍스트를
  문장에 굽는 용도이고 검색은 순수 벡터 유사도 — 계층 주소가 벡터에 미리 들어있어
  결과적으로 올바른 노드에 떨어짐.
- **행/열 트리**: 같은 알고리즘을 축만 바꿔 두 번 적용, 표현상 루트 하나로 묶음.
- **데이터 셀이 트리에 없는 이유**: 셀은 부모가 둘(행·열)이라 트리에 못 들어감.
  트리 = 주소 체계, 셀 = 두 경로의 교차점에서 주소를 부여받는 값.
- **문장 정확도 지표**: 문장 exact match의 P/R/F1(경계 오류의 spurious/dropped를
  각각 P/R이 잡음) + 오류 원인 분해(값/행/열) + 경계 정확도 + 주소충돌 수.
