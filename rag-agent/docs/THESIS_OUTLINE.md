# 석사 학위논문 골격 — 표 RAG에서 구조보존 직렬화와 수치 질의응답

> 대상: 국내 석사 학위논문. 본 문서는 지도교수 보고용 골격 + 현재까지의 실측치를
> 한 곳에 모은 초안이다. 모든 수치는 `rag-agent/results/codegen/`의 실제 산출물.

## 제목(안)
**"검색증강생성(RAG)에서 표의 구조 보존이 수치 질의응답에 미치는 영향:
계층 헤더경로 직렬화의 효과와 한계에 관한 실증 연구"**

## 한 줄 요약(초록 씨앗)
표를 RAG에 넣을 때 청킹·평탄화로 **헤더 구조가 소실되면 수치 계산이 무너진다.**
모든 셀에 **전체 계층 헤더경로를 보존**하면 수치정답 정확도가 회복되며, 이 효과는
**표의 계층성과의 교호작용**이고(평탄표≈0, 계층표에서 큼), **토큰 증가가 아니라 구조
때문**이며, **실제 검색 파이프라인 안에서도** 유지된다. 나아가 구조를 보존하고 나면
**다음 병목은 검색 완전성**이며, 질문→헤더경로 분해가 그 한계(≈0.67)임을 진단한다.

---

## 1. 서론
- 문제: 표는 RAG에서 1순위로 다뤄지나 **수치 질의응답은 미해결** — T²-RAGBench(EACL 2026)
  최고 파이프라인도 oracle 대비 **수치정답 ~30%↓**.
- 가설: 표가 커서 청킹·직렬화될 때 **헤더 구조 소실(F2)**이 **수치 계산(F4)**을 깨뜨린다.
- 기여(석사 수준, 측정·진단형):
  1. 구조보존 직렬화가 수치정확도를 보호함을 **수치-특화 지표 + flat/hier 통제**로 격리 측정
  2. **토큰 통제**로 효과가 구조 기인임을 증명
  3. **실제 검색 파이프라인**에서 효과 유지 + "구조소실이 병목, 검색recall 아님" 규명
  4. **다음 병목(검색 완전성)과 분해기 한계(0.67)** 진단 → 향후 방향 제시

## 2. 관련 연구
- 직렬화/구조표현: Table-Meets-LLM(WSDM24), OHD(2602.01969), STR(2605.31550), TABGR — 계층 보존이 도움(검색 없이, 표 전체 주어진 가정).
- 표-RAG: TableRAG(Google 2410.04739 / Huawei 2506.10380), GTR(2504.01346), STC(2605.00318) — **검색 recall 위주**, 수치 end-to-end 미연결.
- 벤치마크: T²-RAGBench(2506.12071) — 수치 oracle 격차 ~30%(미해결), 방법 미제시.
- **빈자리(본 논문)**: 위 선행은 (a) 검색 없이 직렬화만 보거나 (b) recall만 측정.
  **"청킹/직렬화의 구조소실이 수치 계산을 깨고, 구조보존이 회복함"을 수치정확도로
  end-to-end 측정 + 계층 교호작용 분리**는 미흡.
  (정직한 위치설정: 직렬화 기법 자체는 신규 아님 → 본 논문은 **측정·진단 기여**.)

## 3. 방법 및 실험 설계
- **3단계 직렬화**(구조 보존도): S0 값만 / S1 잎헤더(평탄화) / S2 전체 헤더경로(보존).
- **층화**: flat(WikiSQL, SQL gold) vs hier(HiTab). flat은 **반증 통제**(계층 없으면 효과 없어야).
- **지표**: 수치/정확일치. paired bootstrap 95% CI, seed=42.
- **파이프라인**: 표→행청크→BM25 top-k 검색(조건 간 동일)→직렬화→LLM 계산.
- **통제조건**: header_shuffle = S2와 토큰 동일·경로 오배치(구조만 파괴).
- 모델: llama-3.1-8b-instant, openai/gpt-oss-120b (Groq).

## 4. 결과

> 통계 주의: 유의성은 **McNemar 정확검정**(쌍대 이진의 올바른 검정)으로 보고한다.
> 초기 harness의 percentile-bootstrap CI는 소표본에서 anti-conservative하여 유의성을
> 과대평가했다(`scripts/significance_tests.py`로 재현). **확실한 결과는 4-2·4-3이고,
> 4-1의 교호작용은 방향은 일관되나 n=20에서 검정력이 부족한 "경향"이다.**

### 4-1. 구조보존 × 계층 교호작용 (직접 QA) — *경향(비유의)*
| 직렬화 | flat 8b | hier 8b | flat 120b | hier 120b |
|---|---|---|---|---|
| S0 값만 | 0.70 | 0.10 | 0.55 | 0.10 |
| S1 잎헤더 | 0.85 | 0.20 | 0.80 | 0.25 |
| S2 헤더경로 | 0.90 | 0.50 | 0.85 | 0.45 |

순수 계층보존 효과 **S2−S1** (McNemar exact, n=20):
| 모델 | flat(대조) | hier(효과) |
|---|---|---|
| 8b | +0.05 (p=1.00, 올바르게 null) | +0.30 (**p=0.070, 비유의/경계**) |
| 120b | +0.05 (p=1.00) | +0.20 (**p=0.125, 비유의**) |
→ flat은 올바르게 null. hier은 방향 일관(8b+0.30, 120b+0.20)이나 **유의성 미달**.
정직 서술: *"8b에서 경향, 120b에서 비유의 — 표본(n=20)이 작아 검정력 부족."*
**"두 모델 재현/유의"라고 쓰면 과장이다.**

### 4-2. 토큰 통제 (효과는 구조 기인) — **확실 (p=0.008)**
8b, hier, n=20: flat_leaf 0.15 / header_path 0.45 / **header_shuffle 0.05**
- header_path − header_shuffle = **+0.40, McNemar p=0.008** (불일치쌍 8:0)
- → 토큰 동일·경로만 오배치 시 정확도가 무너짐 = 효과는 토큰이 아니라 **정확한 구조** 덕.
- **논문의 가장 강한 단일 증거. 맨 앞에 배치.**

### 4-3. 실제 검색 파이프라인 (검색이 들어가도 유지) — **확실 (p=0.002)**
8b, hier, n=25, chunk=4행, top_k=3: flat_values 0.08 / flat_leaf 0.20 / **header_path 0.60**,
chunk_recall 0.60.
- header_path − flat_leaf = **+0.40, McNemar p=0.002** (불일치쌍 10:0)
- **header_path 정확도(0.60) = 검색천장(0.60)**: gold가 검색되면 구조보존은 거의 100%
  계산 성공, 평탄화는 ~1/3만. **병목은 청크경계 구조소실, 검색recall 아님.**
- → 4-2와 더불어 **확실한 두 번째 증거**(실제 RAG 파이프라인에서 구조보존이 수치를 살림).

### 4-4. 다음 병목 진단 (검색 완전성 · 분해기 한계)
- 구조를 보존하면 다음 한계는 **필요 셀(피연산자)의 완전 회수**.
- 질문→헤더경로 분해(HPIR) 정확도(HiTab dev, n=200, 무LLM):
  | 매처 | ROW | COL | BOTH |
  |---|---|---|---|
  | fuzzy | 0.79 | 0.67 | 0.61 |
  | embedding(bge-small) | 0.81 | 0.74 | 0.675 |
  | hybrid(fuzzy∪embed) | 0.82 | 0.74 | **0.685** |
  - 검색예산 늘려도 ~0.635, **하이브리드·임베딩·쿼리확장으로도 0.69에서 천장**(정답
    헤더를 *아예 못 잡음*, 순위/튜닝 문제 아님).
  - → **분해 천장은 근본적.** 따라서 방법론적 개선은 "완벽한 분해"가 아니라
    **"불완전 분해에 견고한 fallback 설계"**(저신뢰 시 전체표 청크로 degrade)가 정답.
    (향후: 더 강한 분해 = 별도 연구; 본 논문은 천장을 *측정·정량화*하고 fallback을 제시.)

## 5. 한계 및 향후 연구 (정직하게)
- **통계적 검정력 부족**: 핵심 교호작용(4-1)은 n=20에서 McNemar p=0.07~0.125로 **비유의**.
  방향은 일관하나 "유의/재현"으로 쓰면 안 됨. **확실한 건 4-2(p=0.008)·4-3(p=0.002)뿐.**
  → n을 40~80으로 키워야 4-1을 유의로 끌어올릴 수 있음(8b/120b로 가능).
- **직렬화 기법 자체는 신규 아님**(OHD/STR 등). 본 논문 기여는 측정·진단.
- **헤더경로 분해 0.67 천장** → 피연산자-완전 검색의 단일실패점.
- **금융 도메인 외적타당도 미검증**: FinQA/T²-RAGBench는 본 환경의 무료 LLM
  토큰한도로 실행 불가(8b는 다단계 재무계산 바닥, 대형모델은 일일한도 정체).
  → 적정 compute 확보 시 FinQA에서 oracle 격차의 **문서수준 vs 셀수준 분해** 측정이 향후 핵심.
- 표본 규모(n=20~25), 모델 2종 → 확대 필요.

## 6. 결론
표 RAG의 수치 질의응답에서 **구조 보존이 인과적으로 중요**하며, 그 효과는 계층성과의
교호작용이고 토큰이 아닌 구조 기인이며 실제 검색에서도 유지된다. 구조 보존 이후의
병목은 검색 완전성으로 이동하고, 단순 헤더경로 분해는 그 해결의 상한(≈0.67)이다.
